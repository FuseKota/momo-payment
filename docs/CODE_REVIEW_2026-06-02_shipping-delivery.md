## 1. 実装意図の把握

### 目的
店頭一律ではなく、配送先に応じた**正確な送料**を提示し、顧客が**お届け希望日・時間帯**を指定できるようにする。冷凍食品ECとして配送品質・課金妥当性を高める。

### アプローチ
- `src/lib/shipping/` に純粋関数モジュールを新設（`zones.ts` / `calc.ts` / `time-slots.ts`）。
  - 都道府県 → 配送地帯（13区分）→ 佐川急便60サイズ運賃表 + 箱代¥200 で送料算出。
  - 都道府県ごとの `transit`（お届け日数）+ 店側準備 `PREP_DAYS=2` で最短お届け日を算出。お届け希望日は「最短日 〜 注文日+14日」。
- サーバ（`route.ts`）が送料・地域・日付を再計算・検証し、Stripe Checkout の line item に送料行を追加。
- クライアント（カート / チェックアウト）は同じ純関数を用いて表示し、未確定時は「お届け先により変動」を表示。
- `orders` に `delivery_date` / `delivery_time_slot` 列を追加（migration 00017、CHECK 制約付き）。メール・管理画面・マイページ・完了画面に表示を配線。

### 影響範囲
カート / 配送チェックアウト / 注文作成API / Stripe Webhook / 確認メール / 管理注文詳細 / マイページ注文詳細 / 完了画面 / 入力バリデーション / DBスキーマ / 型定義。SHIPPING 経路に集中し、PICKUP 経路への影響はなし。

## 2. 代替案との比較・トレードオフ

| 案 | 概要 | 評価 |
|----|------|------|
| **採用案** | 都道府県→地帯→運賃表（60サイズ固定）+ 箱代。純関数モジュール | 実装コスト中・保守性高・MVPとして妥当 |
| 代替案A | 固定送料（旧 `SHIPPING_FEE_YEN` 一律） | 単純だが地域差を無視。遠方は赤字・近郊は割高 |
| 代替案B | 商品サイズ/重量の積み上げで実サイズ算出 | 正確だが商品マスタにサイズ情報が必要・複雑 |
| 代替案C | 配送業者の運賃APIをリアルタイム取得 | 最も正確だが外部依存・遅延・障害点が増える |
| 代替案D | 運賃表をDB化し管理画面で編集 | 運用柔軟だが今回の要件には過剰 |

| 観点 | 採用案 | 代替案A | 代替案B/C | 代替案D |
|------|:--:|:--:|:--:|:--:|
| 実装コスト | ○ | ◎ | △ | △ |
| 保守性 | ○ | ◎ | △ | ○ |
| 課金精度 | ○ | ✕ | ◎ | ○ |
| 外部依存 | ◎ | ◎ | ✕(C) | ◎ |

**総合判断**: 採用案は妥当。「60サイズ固定 + 箱代」という割り切りはMVPとして合理的で、純関数を client/server で共有し計算の一貫性を担保している点が良い。運賃表を後から差し替えやすい構造（`ZONE_TO_BASE_FEE` / `PREF_DATA`）も評価できる。

## 3. 採用案の評価

### 良い点
- **収益整合**: 算出送料が Stripe Checkout の line item に正しく加算され（`route.ts`）、`idempotencyKey` で二重課金も防止。サーバが DB 価格 + pref から再計算するため、クライアント表示が改ざんされても課金額はサーバが正。
- **テスト保護**: `src/lib/shipping/__tests__/calc.test.ts`（24ケース）が地帯別送料・ルックアップ正規化・リードタイム・日付範囲・閏年/月跨ぎ/年跨ぎ・`isValidYmd` を網羅。金額に直結するロジックが回帰検出可能。
- **TZ安全**: 日付計算を UTC 固定で行い、「今日」のみ `Intl.DateTimeFormat('en-CA', Asia/Tokyo)` で JST 取得。タイムゾーンずれを回避。
- **ルックアップ堅牢性**: 全角空白除去・接尾辞（都/府/県）両対応・北海道特例。**全47都道府県を網羅**。
- **多層防御**: 時間帯は `DELIVERY_TIME_SLOTS`（TS）→ Zod `z.enum` → DB CHECK 制約 で三重に制限。メール HTML は `escapeHtml` 済み・送料行は固定文字列で注入面なし。
- **一貫表示**: 確認メール / 管理画面 / マイページ / 完了画面 すべてで delivery 情報を表示。地域検証は schema superRefine + route の二重。

### 懸念点
- カート段階の合計表示が送料を除外（後述 MEDIUM）。
- リードタイムが暦日加算で営業日/祝日休配を非考慮（後述 MEDIUM）。
- 沖縄運賃がプレースホルダ（後述・要運用判断）。
- 送料機能と無関係な変更・ドキュメントが同一作業ツリーに混在（後述・コミット衛生）。
