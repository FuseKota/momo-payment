# UIエラーメッセージ 調査・設計書

> 目的: 「どこで・何のエラーが起こりうるか」を網羅的に洗い出し、各エラーに対して表示すべきメッセージ（ja / zh-tw / en）を定義する。
> 調査方法: 12サブシステムの並列カタログ化 + 3レンズ（握り潰し / 抜けた失敗モード / i18n整合性）の批判的レビュー。
> 作成日: 2026-06-24

---

## 0. サマリー（全体像）

### 現状の3つの構造的問題

1. **生エラーコードの露出** — API が返す英語 snake_case トークン（`variant_out_of_stock` `payment_session_failed` `rate_limit_exceeded` `validation_error` `Internal server error` 等）が、フォールバック（`data.error` / `err.error` 直出し）でそのまま Alert / Snackbar に表示される。i18n を通らないため zh-tw / en でも英語が出る。
2. **失敗が「空状態」に化ける** — 取得失敗（401/500/ネットワーク断）を握り潰し、`?? []` で空配列にして「商品がありません」「ニュースはまだありません」「開催情報はまだ公開されていません」と表示。ユーザー/管理者は「壊れた」のか「本当に0件」なのか区別できない。
3. **ステータス未区別** — 429（混雑）/ 403（CSRF/権限）/ 401（セッション切れ）/ オフライン を、すべて `common.unexpectedError`（「予期しないエラーが発生しました」）か生トークンに丸める。受け皿となる共通 i18n キーが存在しない。

### 良好な点（維持）

- **i18nキーは3言語完全整合** — ja / zh-tw / en に純粋なキー欠落バグはゼロ（`complete.honorific` の en='' のみ意図的）。
- **郵便番号検索 / not-found(404) / 監査ログ取得** は失敗と空状態を区別し、3言語ローカライズ済みの良いお手本。
- チェックアウトの**業務系9コード**（温度帯混在/配送不可/対応外地域/お届け日範囲外 等）は i18n マップ済み。

## 1. 推奨する共通エラー基盤（★最優先・全対応の土台）

個別画面を直す前に、まず共通の受け皿を用意する。これが無いと同じ文言が各所に散らばり不統一になる。

### 1-1. 命名規約（3層）

| 層 | 名前空間 | 用途 | 例 |
|----|---------|------|----|
| 横断ステータス系 | `common.errors.*` | 429/403/401/オフライン/5xx 等、画面に依らない共通失敗 | `common.errors.rateLimited` |
| 画面固有の業務エラー | `<feature>.errors.*` | その画面でしか起きない業務エラー | `checkoutShipping.errors.outOfStock` |
| フィールド検証 | `<feature>.validation.*` | 入力項目ごとの検証メッセージ（既存） | `addresses.validation.phoneInvalid` |

### 1-2. 新設する `common.errors.*`（8キー・3言語）

| キー | ja | zh-tw | en |
|------|----|----|----|
| `networkError` | 通信に失敗しました。インターネット接続をご確認のうえ、もう一度お試しください。 | 連線失敗。請確認您的網路連線後再試一次。 | Connection failed. Please check your internet connection and try again. |

### 1-3. 共有マッパー `mapApiErrorToKey(status, errorToken)`

全 `fetch` 呼び出しで使い回す。HTTPステータス + error トークンを `common.errors.*` または画面固有キーに変換する純関数。

- `!navigator.onLine` または `fetch` の `TypeError` → `common.errors.offline` / `networkError`
- `429` → `common.errors.rateLimited`（`Retry-After` があれば秒数を埋め込む）
- `401` → `common.errors.sessionExpired`（+ `/login` 誘導）
- `403 invalid_origin` → `common.errors.forbidden`
- `5xx` / 未知トークン → `common.errors.serverError`（**生トークンは絶対に表示しない**）
- 既知の業務トークン → 画面固有キー

**原則: `data.error` / `err.error` / `err.message` を画面に直出ししない。** 必ず i18n キーへフォールバックする。

### 1-4. エラー境界（error.tsx）の整備

| 対応 | 現状 | 推奨 |
|------|------|------|
| `src/app/[locale]/error.tsx` | **不在**（find結果0） | 新設（`'use client'` + `reset`）。`useTranslations('common')` で `errors.somethingWrong` を3言語表示。セグメント単位で局所回復 |
| `src/app/admin/error.tsx` | 不在 | 新設（固定日本語） |
| `src/app/global-error.tsx` | `lang="ja"` 固定・日本語ハードコード | NEXT_LOCALE Cookie でロケール別静的文言マップ + `<html lang>` 切替。真の最終手段に格下げ |
