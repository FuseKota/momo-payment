# Review - 包括的コードレビュー & 自動修正

変更差分に対して適切なレビュースキルを適用し、修正まで一気通貫で行う。

---

## Phase 1: 差分分析

1. `git diff --name-only` で変更ファイル一覧を取得
2. `git diff` で変更内容を取得
3. 変更ファイルをカテゴリ分類:

| 拡張子/パス                        | カテゴリ   |
| ---------------------------------- | ---------- |
| `.tsx`, `.jsx`, `.vue`, `app/`配下 | frontend   |
| `.ts`（上記以外）, `.py`, `api/`   | backend    |
| `.sql`, `prisma/`, `drizzle/`      | database   |
| `infra/`, `terraform/`, `.yaml`    | operations |

---

## Phase 2: スキル発動 & レビュー

カテゴリに応じて以下のスキルを **Skill tool で発動**:

| カテゴリ   | 発動スキル                                            |
| ---------- | ----------------------------------------------------- |
| frontend   | `gen-frontend-checklist` + `gen-security-checklist`   |
| backend    | `gen-backend-checklist` + `gen-security-checklist`    |
| database   | `gen-database-checklist` + `gen-security-checklist`   |
| operations | `gen-operations-checklist` + `gen-security-checklist` |
| 複合       | 該当するすべてを順次発動                              |

### 全カテゴリ共通で発動

| スキル             | 目的                                   |
| ------------------ | -------------------------------------- |
| `gen-naming-guide` | 命名規則チェック（変数・関数・クラス） |

**命名チェック項目**（gen-naming-guide より）:

- [ ] 名前だけ見て処理の流れが追えるか？
- [ ] boolean が自然言語として読めるか（is/has/can）？
- [ ] 曖昧語（data/info/value/item/result）が放置されていないか？
- [ ] 「and」「or」「with」が名前に含まれていないか？
- [ ] 単位が曖昧な数値がないか（timeoutMs等）？

---

## Phase 3: 修正ポイント集約（優先度分類）

全スキルのレビュー結果から修正が必要な項目を優先度別に整理:

| 優先度       | 条件                                         | 対応         |
| ------------ | -------------------------------------------- | ------------ |
| **CRITICAL** | セキュリティ脆弱性、データ損失の可能性       | 即時修正必須 |
| **HIGH**     | バグ、クラッシュ、重大なパフォーマンス問題   | PR前に修正   |
| **MEDIUM**   | ベストプラクティス違反、軽微なパフォーマンス | 推奨修正     |
| **LOW**      | スタイル、命名、ドキュメント改善             | 任意         |

### 追加チェック項目

#### パフォーマンス

- [ ] N+1クエリがないか
- [ ] 不要な再レンダリングがないか
- [ ] 大きなバンドルサイズの追加がないか
- [ ] メモ化（useMemo/useCallback）の適切な使用

#### コード品質

- [ ] 重複コードがないか
- [ ] 関数の責務が単一か
- [ ] エラーハンドリングが適切か
- [ ] 型定義が適切か（any/unknown の濫用なし）

---

## Phase 4: 修正計画 & 実行

1. 修正ポイントを依存関係でソート
2. 各修正の計画を提示
3. ユーザー確認後、修正を実行
4. 修正後に再度 `git diff` で変更を確認

---

## 出力形式（レビューレポート）

```
## レビューレポート

### 対象ファイル
- file1.ts
- file2.tsx

### 指摘事項

#### CRITICAL (即時修正必須)
| # | ファイル | 行 | 問題 | 修正案 |
|---|----------|-----|------|--------|
| 1 | xxx.ts | 42 | SQLインジェクション | パラメータ化クエリを使用 |

#### HIGH (PR前に修正)
| # | ファイル | 行 | 問題 | 修正案 |
|---|----------|-----|------|--------|
| 2 | xxx.ts | 78 | nullチェック漏れ | Optional chainingを追加 |

#### MEDIUM (推奨修正)
| # | ファイル | 行 | 問題 | 修正案 |
|---|----------|-----|------|--------|
| 3 | xxx.tsx | 15 | useMemo推奨 | 計算結果をメモ化 |

#### LOW (任意)
| # | ファイル | 行 | 問題 | 修正案 |
|---|----------|-----|------|--------|
| 4 | xxx.ts | 5 | 命名改善 | `data` → `candidates` |

### サマリー
| 優先度 | 件数 |
|--------|------|
| CRITICAL | 0 |
| HIGH | 1 |
| MEDIUM | 2 |
| LOW | 1 |

### 承認判定
**APPROVE** / **REQUEST CHANGES**
```

---

## 承認基準

| 判定                | 条件                        |
| ------------------- | --------------------------- |
| **APPROVE**         | CRITICALとHIGHが0件         |
| **REQUEST CHANGES** | CRITICALまたはHIGHが1件以上 |

※ MEDIUMとLOWは承認判定に影響しないが、修正を推奨

---

## 注意事項

- 変更がない場合は「変更差分がありません」と報告
- ステージされた変更とステージされていない変更の両方を対象
- 修正実行前に必ずユーザー確認を取る
- 修正後は `/check` → `/commit` でコミット準備を促す
