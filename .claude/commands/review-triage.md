# Review Triage - 対話型レビュートリアージ

直前の会話で出力されたレビュー結果（`/review` や `/pr-review` 等）を受け取り、指摘を1件ずつ解説しながらユーザーの判断を収集する。最後に判断結果を集約した修正計画を作成する。

**前提**: このコマンドはレビュー結果が既に会話内に存在する状態で呼ばれる。レビュー自体は実行しない。

---

## Phase 1: レビュー結果の解析

直前の会話コンテキストからレビュー結果を読み取る。

1. 会話内のレビューレポートから指摘事項を抽出
2. 各指摘の優先度（CRITICAL / HIGH / MEDIUM / LOW）、ファイル、行番号、問題、修正案を構造化
3. 指摘が0件の場合は「指摘事項なし。修正計画の作成は不要です」と報告して終了

---

## Phase 2: サマリー提示

指摘件数のサマリーを提示し、トリアージの開始を宣言:

```
## トリアージ開始

| 優先度 | 件数 |
|--------|------|
| CRITICAL | X件 |
| HIGH | X件 |
| MEDIUM | X件 |
| LOW | X件 |
| **合計** | **X件** |

1件ずつ解説します。各指摘に対して方針を選択してください。
```

---

## Phase 3: 1件ずつ対話型トリアージ

CRITICAL → HIGH → MEDIUM → LOW の順に、1件ずつ以下を実行する。

### 3.1 指摘の解説

各指摘について以下を出力:

```
---
### 指摘 [現在の番号] / [全体の件数]  [優先度ラベル]

**ファイル**: `path/to/file.ts:行番号`
**問題**: [問題の概要]

#### 解説
[なぜこれが問題なのか、具体的なリスクや影響を平易に説明]

#### 該当コード
[問題のコードを Read で取得して引用。前後数行のコンテキスト付き]
```

### 3.2 修正方針の提案

指摘の内容に応じて **2〜4つの修正方針** を AskUserQuestion の選択肢として提示する。各選択肢の description にトレードオフを記載する。

方針は問題の性質によって柔軟に変える。以下は方針の例:

#### コード修正系の方針例

- **最小修正**: 問題箇所だけをピンポイントで修正。影響範囲が小さく安全だが、根本解決にならない場合がある
- **根本修正**: 設計やデータフローを含めて修正。根本解決だが影響範囲が広く、追加テストが必要
- **リファクタリング**: 周辺コードも含めて構造を改善。保守性が向上するが工数が大きい
- **代替実装**: 別のアプローチで実装し直す。[具体的な代替案]のメリットを享受できるがリスクもある

#### 対応延期系の方針例

- **スキップ**: 意図的にこのままにする。現状で問題は起きていない、または別の理由で許容する
- **Issue化**: 今回は修正せず GitHub Issue として登録する。`/issues` コマンドの手順に従い、種別・優先度ラベル付きで作成する

### 3.3 方針の提示ルール

1. 問題ごとに具体的な修正方針を考え、その問題に適した選択肢を構成する
2. 各選択肢の description には必ず **トレードオフ**（メリット・デメリット・影響範囲）を記載する
3. 推奨する方針がある場合は選択肢の先頭に置き、label に「(推奨)」を付ける
4. 「スキップ」「Issue化」は常に選択肢に含める必要はない。問題の性質上それが妥当な場合のみ含める
5. ユーザーは常に「Other」で自由記述できるため、想定外の方針にも対応可能

### 3.4 判断の記録

各指摘の判断結果を内部的に記録する:

```
{ 番号, 優先度, ファイル, 行, 問題, 選択した方針, 方針の詳細, ユーザー補足 }
```

---

## Phase 4: 修正計画の作成

全件のトリアージが完了したら、判断結果を集約して修正計画を提示する。

### 4.1 判断結果サマリー

```
## トリアージ結果

| # | 優先度 | 問題 | 選択した方針 |
|---|--------|------|-------------|
| 1 | CRITICAL | [概要] | [方針名] |
| 2 | HIGH | [概要] | [方針名] |
| ... | | | |
```

### 4.2 修正計画（コード修正を伴う項目）

依存関係を考慮して修正順序を決定し、計画を提示:

```
## 修正計画

修正順序は依存関係とリスクを考慮して決定しています。

| 順序 | # | 優先度 | ファイル | 方針 | 修正内容 |
|------|---|--------|----------|------|----------|
| 1 | 1 | CRITICAL | xxx.ts:42 | 根本修正 | [修正内容] |
| 2 | 3 | HIGH | xxx.ts:78 | 最小修正 | [修正内容] |
| ... | | | | | |
```

### 4.3 Issue化リスト（Issue化を選択した項目がある場合）

Issue は `/issues` コマンドの手順に従い、種別ラベル + 優先度ラベルを付与して作成する。

```
## Issue化予定

| # | 優先度 | タイトル | ラベル |
|---|--------|----------|--------|
| 5 | MEDIUM | [問題の要約] | enhancement, priority: medium |
```

### 4.4 スキップリスト（スキップを選択した項目がある場合）

```
## スキップ（対応なし）

| # | 優先度 | 問題 | 理由 |
|---|--------|------|------|
| 4 | LOW | [問題] | [ユーザーの判断理由] |
```

---

## Phase 5: 実行確認

修正計画の提示後、AskUserQuestion で最終確認:

| 選択肢           | 動作                                     |
| ---------------- | ---------------------------------------- |
| **計画通り実行** | 修正を順次実行し、Issue化も実施          |
| **修正のみ実行** | コード修正のみ実行、Issue化はスキップ    |
| **計画を修正**   | 修正計画の調整（順序変更・項目追加削除） |
| **中止**         | 何もしない                               |

---

## Phase 6: 実行

ユーザーの最終判断に従い実行:

1. 修正計画に従ってコードを修正
2. 各修正後に `git diff` で変更を確認
3. Issue化対象があれば `/issues` コマンドの手順（種別分析 → 関連コード調査 → ラベル付与 → `gh issue create`）に従って作成
4. 完了報告を出力

### 完了報告

```
## 実行結果

### 修正完了
| # | ファイル | 方針 | 修正内容 | 状態 |
|---|----------|------|----------|------|
| 1 | xxx.ts:42 | 根本修正 | [修正内容] | 完了 |

### Issue作成（該当がある場合）
| # | Issue番号 | タイトル | ラベル |
|---|-----------|----------|--------|
| 5 | #123 | [タイトル] | enhancement, priority: medium |

次のステップ: `/check` → `/commit`
```

---

## 注意事項

- このコマンドはレビューを実行しない。既に会話内にあるレビュー結果を対象とする
- CRITICAL の指摘に対して「スキップ」が選択された場合はリスクを警告する
- 1件ずつの解説では、該当コードを実際に Read して正確な内容を提示する
- ユーザーの判断を尊重し、強制的な修正は行わない
- Issue作成時は `/issues` コマンドの手順に従い、種別ラベル（bug/enhancement）+ 優先度ラベル（priority: high/medium/low）を必ず付与する
