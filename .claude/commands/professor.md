# 教授レビュー

目的は、開発者が**自分の書いたコードを本当に理解しているか**を確認し、理解を深めさせることです。

## 対象ファイルの決定ロジック

**引数**: $ARGUMENTS

### 1. `-r` が指定された場合

- プロジェクト全体から**ランダムに1ファイル**を選択
- `src/` 配下の `.ts`, `.tsx` ファイルを対象とする
- `*.test.ts`, `*.spec.ts`, `*.d.ts` は除外

### 2. 引数なし、または引数が空の場合

以下の優先順で対象を決定:

1. **未コミットの変更ファイルがある場合**
   - `git status` で変更中のファイル（staged + unstaged）を取得
   - その中から**ランダムに1ファイル**を選択
   - 「あなたが今作業中のこのファイルについて質問しますね」と前置き

2. **未コミットの変更がない場合**
   - `git log -1 --name-only` で直近コミットの変更ファイルを取得
   - その中から**ランダムに1ファイル**を選択
   - 「直近のコミットで変更したこのファイルについて質問しますね」と前置き

### 3. ファイルパスが直接指定された場合

- 指定されたファイルを対象とする

## 質問の生成ルール

### 質問数

- 3〜5個（コードが短い場合は少なくてよい）

### 質問の観点（該当する場合のみ）

1. **命名の意図**: 変数名・関数名・クラス名の選定理由
2. **責務の分離**: なぜこの関数/クラスがこの処理を担当するのか
3. **エッジケース**: 境界値や例外時の挙動
4. **設計判断**: なぜこのアプローチを選んだのか、他の選択肢は？
5. **依存関係**: この変更が他の部分に与える影響
6. **テスト観点**: どうやってこのコードをテストするか
7. **パフォーマンス**: 処理効率やメモリ使用の考慮
8. **セキュリティ**: 潜在的なリスクはないか

### 重要

- 必ず**具体的なコード箇所**（変数名、関数名、行の内容など）を引用して質問すること
- 汎用的な質問（「エラーハンドリングは？」など）ではなく、このコード特有の問題を指摘すること
- コードの言語・フレームワーク・文脈を理解した上で質問すること
- **答えを教えるのではなく、考えさせる質問をすること**

## 出力フォーマット

```
## 📚 対象ファイル: `[ファイルパス]`

[選択理由: 未コミット変更 / 直近コミット / ランダム選択 / 指定]

---

### Q1. [具体的なコードを引用した質問]
> 該当コード: `コードスニペット`

**意図**: この質問で確認したいこと

### Q2. [具体的なコードを引用した質問]
> 該当コード: `コードスニペット`

**意図**: この質問で確認したいこと

...

---

💡 **ヒント**: 答えに詰まったら、そのコードを書いたときの判断を思い出してみてください。
   説明できない部分は、理解が曖昧な可能性があります。
```

## 実行手順

1. 引数を解析し、対象ファイルの決定ロジックに従ってファイルを選択
2. 選択したファイルを `Read` ツールで読み取る
3. コードの言語・フレームワーク・目的を理解する
4. 上記の観点から質問を生成する
5. 指定フォーマットで出力する

## 注意

- このコマンドは**テストではなく学習ツール**です
- 開発者が「なんとなく動いた」で終わらせず、意図を持ってコードを書くことを促します
