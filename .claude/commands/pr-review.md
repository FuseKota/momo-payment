# PR-Review: 部下のPR包括的レビュー

部下が作成したPRの変更差分を確認し、実装意図の把握・代替案との比較・トレードオフ分析を行う。

---

## Phase 1: PR情報の取得

### 1.1 PR番号の確認

引数としてPR番号が渡された場合はそれを使用。渡されない場合は現在のブランチから推定またはユーザーに確認。

```bash
# PR情報を取得
gh pr view <PR番号> --json number,title,headRefName,baseRefName,body,author,files

# 変更ファイル一覧
gh pr diff <PR番号> --name-only

# 変更差分
gh pr diff <PR番号>
```

### 1.2 ブランチ切り替え（必要に応じて）

```bash
git fetch origin <headRefName>
git checkout <headRefName>
```

---

## Phase 2: 変更内容の分析

### 2.1 ファイルカテゴリ分類

| 拡張子/パス                        | カテゴリ   |
| ---------------------------------- | ---------- |
| `.tsx`, `.jsx`, `.vue`, `app/`配下 | frontend   |
| `.ts`（上記以外）, `.py`, `api/`   | backend    |
| `.sql`, `prisma/`, `drizzle/`      | database   |
| `middleware.ts`                    | middleware |
| `infra/`, `terraform/`, `.yaml`    | operations |

### 2.2 変更内容の詳細読解

各変更ファイルを読み、以下を把握:

- 何が変更されたか（What）
- なぜ変更されたか（Why）- PRのbodyやコミットメッセージから推定
- どのように実装されたか（How）

---

## Phase 3: 実装意図の把握

以下の観点で実装意図を整理:

### 3.1 目的

- この変更が解決しようとしている問題は何か？
- 関連するIssue/要件は何か？

### 3.2 アプローチ

- 採用された実装方針は何か？
- なぜこのアプローチが選ばれたと考えられるか？

### 3.3 影響範囲

- この変更が影響する機能・コンポーネントは何か？
- 副作用やリグレッションのリスクはあるか？

---

## Phase 4: 代替案の検討

### 4.1 代替案のブレインストーミング

この実装ではない場合、他にどのような選択肢があったか列挙:

```
代替案A: [概要]
代替案B: [概要]
代替案C: [概要]
```

### 4.2 各案の評価軸

| 評価軸           | 説明                           |
| ---------------- | ------------------------------ |
| 実装コスト       | 開発工数、複雑さ               |
| 保守性           | 将来の変更容易性、理解しやすさ |
| パフォーマンス   | 実行速度、リソース消費         |
| スケーラビリティ | 要件変更・拡張への対応力       |
| セキュリティ     | 脆弱性リスク                   |
| 一貫性           | 既存コードベースとの整合性     |

```markdown
| パフォーマンス   | [評価] | [評価]  | [評価]  |
| スケーラビリティ | [評価] | [評価]  | [評価]  |
| セキュリティ     | [評価] | [評価]  | [評価]  |
| 一貫性           | [評価] | [評価]  | [評価]  |
```

### 5.2 採用案の妥当性評価

- **良い点**: 採用案の強み
- **懸念点**: 採用案の弱み・将来的なリスク
- **総合判断**: 採用案は妥当か / より良い選択肢があったか

## Phase 6: コードレビュー

カテゴリに応じたレビュースキルを発動:

| カテゴリ   | 発動スキル                                          |
| ---------- | --------------------------------------------------- |
| frontend   | `gen-frontend-checklist` + `gen-security-checklist` |
| backend    | `gen-backend-checklist` + `gen-security-checklist`  |
| database   | `gen-database-checklist` + `gen-security-checklist` |
| middleware | `gen-backend-checklist` + `gen-security-checklist`  |
| 複合       | 該当するすべてを発動                                |

全カテゴリ共通: `gen-naming-guide`

## 承認基準

| 判定                | 条件                        |
| ------------------- | --------------------------- |
| **APPROVE**         | CRITICALとHIGHが0件         |
| **REQUEST CHANGES** | CRITICALまたはHIGHが1件以上 |

---

## 注意事項

- 単なるコード指摘だけでなく、「なぜこの実装を選んだか」の理解を重視
- 代替案は机上の空論ではなく、現実的に検討可能なものを挙げる
- トレードオフ分析では、採用案の弱みも公平に評価する
- 部下の成長につながるフィードバックを意識する
